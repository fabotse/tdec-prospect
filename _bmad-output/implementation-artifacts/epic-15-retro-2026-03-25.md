## Desafios e Aprendizados

### 1. Apollo api_search retorna dados LIMITADOS (15.4)

**Problema:** O endpoint Apollo `/v1/mixed_people/api_search` retorna dados de prospecao, nao dados completos. `email`, `linkedin_url` e `phone` vem como `null`. `last_name` vem ofuscado ("Sil***a").

**Impacto:** AC #3 da Story 15.4 precisou ser adaptado — em vez de exibir email real, exibe badges de disponibilidade (`hasEmail`, `hasDirectPhone`).

**Aprendizado:** APIs de prospecao frequentemente separam "busca" de "enriquecimento". O dado completo so vem apos enrichment (que consome credits adicionais). Planejar stories considerando essa limitacao.

### 2. Erros TypeScript pre-existentes quebravam build (entre 15.4 e 15.5)

**Problema:** Commit `49c5ff2` (fix: corrigir erros TypeScript que quebravam build de producao) foi necessario entre stories. Erros em arquivos nao-relacionados ao epic impediam `npm run build`.

**Causa raiz:** Mesmo problema da Epic 14 (story 14.3) — erros TS acumulados de epics anteriores nao detectados ate rodar build completo.

**Aprendizado:** `npm run build` deve ser executado no INICIO de cada epic (nao apenas ao final de cada story) para detectar debt TS pre-existente.

### 3. Arrays enormes na response theirStack (15.2)

**Problema:** `technology_slugs` e `technology_names` podem ter 3000+ items por empresa. `keyword_slugs` pode ter 7000+ items. Exibir raw na UI causaria problemas de performance e UX.

**Solucao:** Filtrar `technologies_found` para mostrar apenas as tecnologias buscadas (com confidence badge), ignorando os arrays brutos.

**Aprendizado:** APIs de dados technograficos retornam volume massivo de metadata. Sempre filtrar no service/transform layer antes de enviar para a UI.

### 4. Metadata de source sem coluna dedicada (15.5)

**Problema:** O schema de `leads` nao tem colunas para `source` ou `sourceTechnology`. Adicionar migration seria scope creep.

**Solucao:** Reutilizou `lead_interactions` com `type='import'` e `content` JSON contendo `{"source":"theirStack + Apollo","technology":"Netskope","createdVia":"technographic-prospecting"}`.

**Aprendizado:** Tabelas de interacoes/log sao mecanismos flexiveis para metadata sem alterar schema principal. Padrão valido para features aditivas.
