## Deferred from: code review of story-20.3 (2026-06-16)

> Review adversarial 3 camadas (Blind Hunter / Edge Case Hunter / Acceptance Auditor), modo YOLO. 0 bug HIGH. ACs 1/2/3 verificados independentemente pelo Acceptance Auditor (sem scope creep p/ 20.4/20.5). 2 patches aplicados nesta review (test-hardening dos predicados de segurança em `updateMemberRole` + `console.error` no caminho de 0 linhas). Defers abaixo são reais mas pré-existentes/arquiteturais — roteados para a **20.5** (auditoria defense-in-depth).

- **Invariante "≥1 admin" vive só na camada de aplicação (TOCTOU + fail-open + sem garantia no banco).** A proteção do último admin em `updateMemberRole` ([team.ts:441-459](../../src/actions/team.ts#L441-L459)) é um check-then-update **não transacional**: (a) **TOCTOU** — dois admins rebaixados concorrentemente passam ambos pelo `count===2` e o tenant fica com 0 admins; (b) **fail-open** — se a query de contagem falhar e retornar `count: null`, `null === 1` é `false` → o rebaixamento prossegue; (c) a RLS (00054) só garante "o ator é admin", **não** "resta ≥1 admin". Mesmo padrão já existe em `removeTeamMember`/`isOnlyAdmin` (pré-existente, não regressão da 20.3) e a spec mandou espelhá-lo. Fix robusto = trigger/constraint no banco ou contagem em transação serializável. Severidade: MEDIUM (concorrência/edge; baixíssima probabilidade com ~4 usuários). **Dono: Story 20.5.** Fonte: Blind Hunter + Acceptance Auditor, review 20.3.
- **Auto-rebaixamento deixa o estado de admin do cliente obsoleto (`useUser`).** Quando há ≥2 admins, a UI habilita "Alterar Função" na **própria** linha do usuário ([TeamMemberList.tsx:95-96](../../src/components/settings/TeamMemberList.tsx#L95-L96)) e o servidor permite o auto-rebaixamento (count=2). Mas a mutation de papel só invalida `["team","members"]` e `["team","isOnlyAdmin"]` ([use-team-members.ts:108-114](../../src/hooks/use-team-members.ts#L108-L114)); o `useUser` mantém `role`/`isAdmin` num singleton de módulo **fora** do TanStack Query, então `AdminGuard`/sidebar continuam mostrando UI de admin até reload/re-login. **Sem brecha de segurança** (o servidor barra ações privilegiadas), só estado de UI falso. Severidade: MEDIUM (UX, raro). **Dono: Story 20.5** (varredura SDR/superfícies admin é o lugar natural). Fonte: Edge Case Hunter, review 20.3.
- **Policy RLS de UPDATE é row-level (não restringe a coluna `role`).** A 00054 concede UPDATE de **qualquer coluna** de perfis do mesmo tenant ao admin — RLS no Postgres não restringe colunas. Mitigado: a única caller (`updateMemberRole`) escreve apenas `{ role }`, e a policy é **simétrica** à de DELETE de admin já existente (00009). Endurecimento opcional (trigger que limite as colunas alteráveis) cabe na auditoria. Severidade: LOW. **Dono: Story 20.5** (nota de hardening). Fonte: Blind Hunter, review 20.3.

Follow-up sugerido: decidir se o default deve honrar `prefers-color-scheme` no script inline, ou documentar o default-dark como intencional. Severidade: MEDIUM (UX), sem impacto funcional no logo. Fonte: Blind Hunter + Edge Case Hunter, review 19.1.
