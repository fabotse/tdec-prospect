## O que foi bem

1. **Escopo cirúrgico (19.2).** O risco real nunca foi complexidade — era excesso de zelo. Inventário exato de 7 strings preservou ~50 fixtures de teste, o design-system "TripleD" e o slug técnico `tdec-prospect`. Um find-replace global teria quebrado tudo isso.
2. **Decisão de arquitetura correta (19.1).** CSS puro (`dark:`) em vez de `useTheme()` para evitar FOUC do logo (o snapshot SSR do ThemeProvider é sempre `dark`). Decisão alinhada à memória do projeto e bem documentada nas Dev Notes.
3. **Centralização que paga dividendo.** A constant `BRAND` virou fonte única: 19.1 criou, 19.2 e 19.3 consumiram. Base pronta para o white-label real.
4. **Reviews adversariais funcionaram.** Nas 19.1/19.3, o Blind Hunter levantou hipótese de FOUC do logo e o Edge Hunter refutou com acesso ao código — o processo se autocorrigiu e a reconciliação ficou documentada.
5. **Investigação real, não suposição (19.2).** Descobriu-se que a migration `00038` já havia desativado os prompts globais do DB → a "armadilha do DB override" estava neutralizada. Estado do sistema foi verificado, não assumido.
6. **Composição limpa entre stories.** Decisões de fronteira explícitas (19.1 reservou metadata/favicon/OG para a 19.3; 19.2 documentou "Fronteira com a 19.3") evitaram overlap.

## O que pode melhorar

1. **Retrabalho na 19.1.** Logo foi primeiro pro Header (quebrou 18 testes: `TypeError: Invalid URL` por mock de `window.location` + `next/image`), depois movido pro Sidebar por feedback do usuário, com Header revertido. O feedback de UI (posição, tamanho `h-12`→`h-24`) só veio **depois** da implementação.
2. **Decisões deixadas para o meio do dev.** Várias "Open Questions" / "confirmar com usuário" (opção A/B do prompt no DB; favicon.ico) surgiram durante a implementação; algumas só fecharam na review de 2026-06-15, ~2 semanas após a implementação (2026-06-01).
3. **Gap de assets.** Imagem OG não existia (gerada na própria story); favicon chegou do cliente no meio. Dependência externa não estava pronta no início.

## Padrões entre stories

- As 3 stories: "100% cosmética, não toca lógica de negócio" — risco = escopo, não complexidade.
- Recorrência do problema `next/image` + mock de `window.location` (19.1) — já é memória do projeto, continua mordendo.
- Consumo progressivo de `BRAND` — boa coordenação inter-story.
- "Armadilhas" bem documentadas nas Dev Notes (favicon default, DB override do prompt) — alta qualidade de context engineering na criação das stories.
